markdown# Watch Party Finder — Project Notes

---

## Project Overview
A website where people can find FIFA watch parties near them.
Shows live scores, venue locations on a map, how busy venues are,
and crowd hype based on local fan demographics.

## Tech Stack
- **Backend:** Python, Flask, SQLAlchemy
- **Database:** PostgreSQL
- **Frontend:** HTML, CSS, JavaScript, Google Maps API
- **External APIs:** Football API (scores), Google Maps (venues), Demographics data
- **Deployment:** Render (backend), Vercel (frontend)

---

## System Design
User opens website

→ Browser sends request to Flask "give me venues"

→ Flask routes it to get_venues() function

→ SQLAlchemy asks PostgreSQL "give me all rows in venue table"

→ PostgreSQL returns 4 venues

→ Flask converts them to JSON

→ Browser receives and displays data

Layers:
Browser (user sees this)

↕ HTTP requests

Flask app (Python code, handles logic)

↕ SQL queries

PostgreSQL (stores data)

## Key Concepts

### Virtual Environment (venv)
Isolates your project's packages from your global Python installation.
Every project gets its own venv so packages don't conflict between projects.
- Create: `python3 -m venv venv`
- Activate: `source venv/bin/activate`
- Confirm active: prompt shows `(venv)`
- Always activate before installing packages or running the app

### .env and python-dotenv
`.env` stores secret values like database URLs and API keys.
Never commit this file — it's in `.gitignore`.
`python-dotenv` reads `.env` and loads values so your app can access them.
Always create a `.env.example` with fake placeholder values so others
know what variables are needed without exposing real secrets.

### requirements.txt
A list of every package your project needs with exact version numbers.
Generated by running `pip freeze > requirements.txt`.
Anyone cloning your project runs `pip install -r requirements.txt`
to get your exact same setup.

### Flask App Factory Pattern
Instead of `app = Flask(__name__)` at the top of a file,
you wrap app creation in a `create_app()` function.
Why: makes the app testable and configurable.
Flask finds it automatically when you run `flask shell` or `flask run`.

### Blueprint
Flask's way of grouping related routes together.
Instead of registering all routes directly on the app,
you group them — venues routes in one blueprint,
match routes in another later.
Registered in create_app() with `app.register_blueprint(venues_bp)`.

### SQLAlchemy
A Python library that lets you interact with your database
using Python objects instead of raw SQL.
Without it: `SELECT * FROM venues WHERE neighbourhood = 'Gastown'`
With it: `Venue.query.filter_by(neighbourhood='Gastown').all()`
`flask-sqlalchemy` is a wrapper that makes SQLAlchemy work with Flask.
- `db = SQLAlchemy()` — create instance outside create_app() so models can import it
- `db.init_app(app)` — bind SQLAlchemy to your Flask app
- `db.create_all()` — create all tables defined in your models

### Session (database)
Think of it as a shopping cart.
You add objects to it, then commit to actually save to database.
If something goes wrong you can roll it back.
- `db.session.add(venue)` — stage a change
- `db.session.commit()` — write everything to database

### jsonify
Converts a Python dictionary or list into a proper JSON response.
JSON is the standard language browsers and APIs use to exchange data.
You can't return Python objects directly — must convert to dicts first,
then pass to jsonify().

### app_context()
Flask needs an active context to know which app and database to use.
Inside a running server this exists automatically.
In scripts you create it manually:
```python
with app.app_context():
    db.create_all()
```

### `__name__`
A built-in Python variable every file has automatically.
Holds the name of the current module.
Flask uses it to know where your project lives on disk.
When you run a file directly its __name__ equals '__main__'.
Always use `Flask(__name__)` — never change this.

### `if __name__ == '__main__'`
Only runs the code inside if you're executing this file directly.
If another file imports it, this block is skipped.
Used in run.py so the server only starts when you run it directly.

### app/__init__.py
```python
# Application factory — creates and configures the Flask app

from flask import Flask
# Imports the Flask class from the flask package

from flask_sqlalchemy import SQLAlchemy
# Imports SQLAlchemy — the translator between Python and PostgreSQL
# flask_sqlalchemy is a wrapper that makes SQLAlchemy work with Flask

db = SQLAlchemy()
# Creates a SQLAlchemy instance OUTSIDE create_app()
# Must be outside because models/__init__.py imports db
# If it were inside create_app(), nothing else could access it

def create_app():
    # The factory function — everything that sets up Flask lives here

    app = Flask(__name__)
    # Creates the Flask application instance
    # __name__ helps Flask know where your project lives on disk

    app.config['SQLALCHEMY_DATABASE_URI'] = 'postgresql://localhost/watchparty_db'
    # Tells SQLAlchemy where your database is
    # postgresql — use the PostgreSQL adapter (psycopg2)
    # localhost — database is on this same machine
    # watchparty_db — the specific database we created
    # On a real server this becomes: postgresql://user:password@server/watchparty_db
    # That's why it will move to .env — never hardcode real passwords

    app.config['SQLALCHEMY_TRACK_MODIFICATIONS'] = False
    # Silences a deprecation warning from SQLAlchemy
    # Track modifications adds overhead — setting False suppresses the warning

    db.init_app(app)
    # Binds the SQLAlchemy instance to this Flask app
    # Now SQLAlchemy knows which app's config to read
    # Reads SQLALCHEMY_DATABASE_URI to find your database

    from app.routes import venues_bp
    # Import is INSIDE create_app() on purpose
    # Avoids circular imports — routes imports from app, app imports from routes
    # Doing it inside the function breaks that cycle

    app.register_blueprint(venues_bp)
    # Registers the venues blueprint with Flask
    # Now Flask knows about all routes defined in venues_bp

    return app
    # Returns the fully configured Flask app
    # run.py receives this and uses it to start the server
```

### app/models/__init__.py
```python
from app import db
# Imports the SQLAlchemy instance from app/__init__.py
# db is the object we use to define columns and query the database

class Venue(db.Model):
# Defines a Python class called Venue
# db.Model means this class represents a database table
# SQLAlchemy reads this and knows how to create/query the venue table

    id = db.Column(db.Integer, primary_key=True)
    # Integer column — uniquely identifies each row
    # primary_key=True — auto increments (1, 2, 3 etc.) automatically

    name = db.Column(db.String(100), nullable=False)
    # String column, max 100 characters
    # nullable=False means required — can't be empty

    address = db.Column(db.String(200))
    # String column, max 200 characters — optional

    neighbourhood = db.Column(db.String(100))
    # String column for neighbourhood name

    latitude = db.Column(db.Float)
    # Float (decimal) for latitude e.g. 49.2756
    # Can't be Integer — would lose precision after decimal point

    longitude = db.Column(db.Float)
    # Float for longitude e.g. -123.1319
    # Negative because west of prime meridian

    capacity = db.Column(db.Integer)
    # Integer for how many people the venue holds

    num_screens = db.Column(db.Integer)
    # Integer for number of TV screens

    opening_hours = db.Column(db.String(50))
    # String because opening hours are text like "4pm - 2am"

    cover_charge = db.Column(db.Float)
    # Float because it's a dollar amount like 10.00

    busyness_score = db.Column(db.Integer)
    # Integer 0-100 representing how busy the venue is
    # Hardcoded for now — will come from Google Places API later

    def __repr__(self):
    # Controls how a Venue object displays when printed
    # Without it: <Venue object at 0x10f3a2c50>
    # With it: <Venue The Penalty Box>
        return f'<Venue {self.name}>'
```

### app/routes/__init__.py
```python
from flask import Blueprint, jsonify
# Blueprint — groups related routes together
# jsonify — converts Python dict/list to proper JSON response

from app.models import Venue
# Imports Venue model so we can query the database

venues_bp = Blueprint('venues', __name__)
# Creates a Blueprint called venues
# 'venues' is the name Flask uses internally
# __name__ helps Flask locate the blueprint

@venues_bp.route('/api/venues')
# Decorator that tells Flask: when someone requests GET /api/venues
# run the function below
# Default HTTP method is GET — used for fetching data

def get_venues():
    venues = Venue.query.all()
    # Asks SQLAlchemy to query ALL rows in venue table
    # Translates to SQL: SELECT * FROM venue
    # Returns a list of Venue Python objects

    venues_list = []
    # Empty list that will hold dictionaries

    for venue in venues:
    # Loops through every Venue object

        venues_list.append({
        # Converts each Venue object to a dictionary
        # Can't jsonify Python objects directly — must be dicts first
            'id': venue.id,
            'name': venue.name,
            'address': venue.address,
            'neighbourhood': venue.neighbourhood,
            'latitude': venue.latitude,
            'longitude': venue.longitude,
            'capacity': venue.capacity,
            'num_screens': venue.num_screens,
            'opening_hours': venue.opening_hours,
            'cover_charge': venue.cover_charge,
            'busyness_score': venue.busyness_score
        })

    return jsonify(venues_list)
    # Converts list of dicts to JSON
    # This is what appears in the browser at /api/venues
    # return is OUTSIDE the for loop — runs after all venues are added
```

### seed.py
```python
from app import create_app, db
from app.models import Venue

app = create_app()
# Create the Flask app so we have database connection

with app.app_context():
# Manually creates Flask context so this script can use the database
# "with" block means context is active for everything inside it

    Venue.query.delete()
    # Deletes all existing venues before seeding
    # Prevents duplicates if you run seed.py multiple times

    venue1 = Venue(name="The Penalty Box", ...)
    # Creates a Venue Python object with all column values
    # Not saved to database yet — just exists in memory

    db.session.add(venue1)
    # Adds venue to the session (staging area)
    # Nothing written to database yet

    db.session.commit()
    # NOW writes everything in the session to PostgreSQL
    # All or nothing — if anything fails, nothing gets saved
```

### create_tables.py
```python
from app import create_app, db
from app.models import Venue

app = create_app()

with app.app_context():
    db.drop_all()
    # Drops all existing tables
    # Only use when you change model columns and table is empty
    # DANGEROUS on real app with real data — deletes everything
    # Remove this line after running once

    db.create_all()
    # Reads all your models and creates tables in PostgreSQL
    # Safe to run multiple times — won't overwrite existing tables
    # Equivalent SQL:
    # CREATE TABLE venue (
    #   id SERIAL PRIMARY KEY,
    #   name VARCHAR(100) NOT NULL,
    #   address VARCHAR(200),
    #   ... etc
    # )

    print("Tables created successfully")
```

## How Flask Gets You The Venues — Full Journey

You run: python run.py

→ run.py calls create_app()
create_app() runs:

→ Flask app created

→ SQLALCHEMY_DATABASE_URI tells SQLAlchemy where database is

→ db.init_app(app) connects SQLAlchemy to Flask

→ venues_bp registered so Flask knows about /api/venues
Server starts at http://127.0.0.1:5000

→ Waiting for requests
You visit http://127.0.0.1:5000/api/venues in browser

→ Browser sends GET request to Flask
Flask sees /api/venues

→ Matches it to get_venues() function in routes/init.py
get_venues() runs:

→ Venue.query.all() fires SQL: SELECT * FROM venue

→ PostgreSQL returns 4 rows

→ Python converts each row to a dictionary

→ jsonify() converts list of dicts to JSON
Flask sends JSON back to browser

→ You see your 4 venues displayed

## Common Errors & Fixes

### `fatal: not a git repository`
Run `git init` inside your project folder first.

### `venv: error: the following arguments are required: ENV_DIR`
Run `python3 -m venv venv` not just `python3 -m venv`.

### `ModuleNotFoundError: No module named 'psycopg2'` in flask shell
Anaconda conflict — flask shell picked up Anaconda's Python instead of venv's.
Fix: use a plain Python script with app_context() instead of flask shell.
Always run `which python` to confirm venv is active.

### `SQLALCHEMY_TRACK_MODIFICATIONS` warning
Add `app.config['SQLALCHEMY_TRACK_MODIFICATIONS'] = False` to create_app().

### `NameError: name 'db' is not defined` in flask shell
Run `from app import db` explicitly before using db.

### Circular import error
Keep `from app.routes import venues_bp` inside create_app() not at the top level.

## Anaconda Conflict Note
Machine has Anaconda installed alongside venv.
Prompt shows `(venv) (base)` — both active at once.
This can cause flask shell to use Anaconda's Python instead of venv's.
Always confirm `which python` points to `watchparty/venv/bin/python`.
When flask shell misbehaves, use a plain Python script instead.
